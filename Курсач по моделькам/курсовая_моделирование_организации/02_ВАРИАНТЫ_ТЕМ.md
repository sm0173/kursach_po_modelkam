## ТЕМА #1 — Динамическое прогнозирование ETA + управление риском срыва SLA

**Статус в проекте:** ✅ в курсаче; **детальный BPMN** в гл. 3

### Проблема (бизнес-формулировка)

Клиенту в цифровом канале обещают время доставки, но при пиковой нагрузке (кухня, курьеры, очередь) **ETA не пересчитывается вовремя** и система **не выбирает управляемый сценарий** — заказ принимают с нереалистичным обещанием или клиент узнаёт о задержке слишком поздно. Итог: срыв SLA, отмены, холодная еда, падение рейтинга.

### Рабочее название процесса для BPMN

> «Совершенствование процесса обработки заказа на доставку при риске превышения обещанного времени доставки в Додо Пицце»

Альтернатива (длиннее): «Управление риском срыва SLA доставки за счёт динамического прогнозирования ETA и выбора альтернативного сценария исполнения заказа».

### SIPOC (черновик)

| | |
|---|---|
| **Триггер** | Клиент оформляет заказ на доставку (адрес + корзина) |
| **Результат** | Заказ принят с реалистичным ETA и исполнен ИЛИ клиент информирован и отказался до оплаты |
| **In scope** | Расчёт/пересчёт ETA, оценка риска SLA, выбор сценария, уведомления клиента, назначение исполнения |
| **Out of scope** | Производство на кухне (детально), маркетинг, оплата (кроме отказа до оплаты) |

### AS IS (реконструкция по открытым данным)

Dodo IS на станции доставки: прогноз времени, учёт транспорта и дорог, подсветка рисковых заказов (жёлтый/красный), маршрутизация. Dodo Brands: order tracking, delivery management, demand forecasting.  
**[ДОПУЩЕНИЕ]:** точные правила customer-facing ETA и поведение при перегрузе публично не раскрыты.

**Источники AS IS:** report.dodopizza.info/delivery, dodobrands.io, отчёты Dodo 2018/2019.

### TO BE — ядро

До подтверждения заказа система считает **реалистичный ETA** с учётом загрузки кухни, очереди, курьеров, времени суток. По уровню риска SLA запускается **дерево решений**:

| Уровень риска | Действие |
|---------------|----------|
| **Низкий** | Обычное принятие заказа с ETA |
| **Средний** | Честный увеличенный ETA + экран выбора (позже / самовывоз) |
| **Высокий** | Операционное дерево (ветки ниже) → отказ до оплаты, если SLA нереален |

После принятия — **проактивный пересчёт ETA** и push при задержке.

### TO BE — ветки (подпункты, не отдельные BPMN)

Эти идеи команды **встроены внутрь** темы #1 как исключительные сценарии:

#### Ветка A: Перегруз точки → альтернативная пиццерия

- При перегрузе обслуживающей точки — проверка **резервной пиццерии**.
- Клиенту: новый ETA, условия, выбор согласиться / отказаться.
- **Не standalone:** нет публичного AS IS; только TO BE-гипотеза + опрос.

#### Ветка B: Кластерное перераспределение курьеров — ⏸ под вопросом

- Заказ готов, локальных курьеров не хватает → проверка **соседней точки в кластере**.
- **Решение команды:** A + C точно; B — включить после этапа 4 или вынести в заключение как перспективу.

#### Ветка C: Проактивные уведомления

- Пересчёт ETA после принятия заказа при событиях (задержка кухни, курьера).
- Push клиенту до того, как он сам заметит опоздание.

### Гипотезы для главы 2

- Часть недовольства связана с **расхождением обещанного и фактического ETA**, а не только с качеством еды.
- Клиенты предпочитают **честный длинный ETA до оплаты** поздней доставке без предупреждения.
- Клиенты готовы к альтернативному сценарию (другая точка, позже) при явном выборе `[проверить опросом]`.

### Эмпирика (без инсайдеров)

- Опрос **40–60** пользователей доставки еды (желательно ≥50% с опытом Dodo).
- **8–12** интервью с теми, кто переживал опоздание / холодный заказ.
- Кодирование **80–120** публичных отзывов (App Store / Google Play): ETA, задержки, холодная еда.

### Метрики эффекта TO BE

ETA MAE/MAPE, on-time %, SLA breach rate, cancellations-before-payment, conversion при высокой загрузке, cold-food complaints, post-order rating.

### Риски на защите

- AS IS частично на допущениях → нужен **assumptions register** и честная формулировка «реконструкция».
- Ветки A/B — TO BE-гипотезы, не «так уже работает у Додо».

### Артефакты курсача

Landscape (5–7 блоков), SIPOC, BPMN AS IS + TO BE, assumptions log, decision tree альтернатив, mockup экрана ETA, опционально — scorecard ETA на синтетических данных.

## ТЕМА #2 — Рекламации + LLM-assisted support

**Статус в проекте:** ✅ в курсаче; **модель TO BE** в гл. 3 (упрощённый BPMN или отдельный параграф)

### Проблема (бизнес-формулировка)

Клиент получил **проблемный заказ** (не тот, холодный, опоздавший). Путь жалобы **непрозрачен и неоднороден**: в чате поддержки слабое NLU → **почти всегда зовётся оператор**; решения по компенсации непредсказуемы; время ответа варьируется.

### Рабочее название процесса для BPMN

> «Совершенствование процесса обработки клиентских рекламаций в Додо Пицце с LLM-помощником для чат-бота и оператора»

### SIPOC (черновик)

| | |
|---|---|
| **Триггер** | Клиент инициировал жалобу (оценка <5 в приложении, чат, звонок) |
| **Результат** | Решение по компенсации + обратная связь клиенту + фидбек на пиццерию |
| **In scope** | Приём обращения, классификация, triage, эскалация, ответ, закрытие кейса |
| **Out of scope** | Производство заказа, доставка (кроме жалоб на них) |

### AS IS (сильная опора на открытые данные)

Официально описанный Dodo flow:
1. Клиент ставит оценку заказу; если <5 — выбирает **типовую проблему**, комментарий, **фото**.
2. Отзыв → система обработки обратной связи.
3. Менеджер пиццерии связывается с клиентом **в течение дня**, решает по корпоративным принципам.

**[ДОПУЩЕНИЕ]:** детальная схема handoff app ↔ чат ↔ contact center публично не раскрыта.

**Источники AS IS:** dodopizzastory.com (order ratings), dodobrands.io (contact center), кейс Tovie AI (Dodo).

### TO BE — ядро

Любое обращение (app / чат / contact center) проходит **AI-assisted triage**:

| Шаг | Что делает система |
|-----|-------------------|
| 1 | Собирает текст, фото, данные заказа |
| 2 | Классифицирует проблему и приоритет (retrieval + rules + LLM) |
| 3 | Делает summary и предлагает **черновик ответа** + варианты из policy matrix |
| 4 | Оператор/менеджер **проверяет** (human-in-the-loop) |
| 5 | Простые кейсы — быстрое закрытие; сложные — эскалация менеджеру |

**LLM = assistant for operator/manager**, не автономный decision maker (риск DPD-антикейса).

### TO BE — подпункты

- **Policy matrix** по типам жалоб (не тот заказ / холодный / опоздание / повреждённая упаковка).
- **Автотриггеры** при повторных жалобах на одну пиццерию → эскалация.
- **Улучшение чата:** NLU + LLM для уточнения намерения до вызова оператора (твоя идея про чат-бот).
- **Guardrails:** платежи, безопасность, крупные компенсации — всегда человек.

### Гипотезы для главы 2

- Значимая доля клиентов **не обращается** с жалобой из-за непонятного пути.
- Операторы тратят время на **рутинную классификацию**, которую можно автоматизировать.
- Клиенты **доверят первичный контакт AI**, если сложные кейсы гарантированно уходят человеку `[проверить опросом]`.

### Эмпирика (без инсайдеров)

- Опрос **20+** клиентов: «обращались с жалобой — что было, как быстро ответили».
- **8–12** интервью по complaint journey.
- Кодирование **80–120** отзывов: «долгий ответ», «непонятный саппорт», «компенсация».

### Метрики эффекта TO BE

First response time, median resolution time, escalation rate, repeat complaint rate по точке, post-case CSAT, accuracy policy recommendation.

### Риски на защите

- TO BE не должен выглядеть как «поставили ChatGPT» — нужны **правила процесса** и human-in-the-loop.
- LLM не утверждает денежные компенсации автономно.

### Артефакты курсача

Landscape блока «послепродажный сервис», SIPOC, BPMN AS IS + TO BE, policy matrix, mockup operator console, опционально — demo triage на 20–30 обезличенных кейсах.
