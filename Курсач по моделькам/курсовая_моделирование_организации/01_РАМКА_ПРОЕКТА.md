## Объект и предмет

| | Формулировка |
|---|--------------|
| **Объект исследования** | Додо Пицца / Dodo Brands — сеть пиццерий с доставкой и food-tech платформой Dodo IS |
| **Сфера деятельности** | Доставка готовой еды в России через цифровые каналы (мобильное приложение, сайт) |
| **Предмет исследования** | Два связанных процесса цифрового клиентского опыта в контуре доставки: (1) **проверка доступности, выбор точки обслуживания и сопровождение срока доставки** при риске срыва норматива; (2) **обработка клиентской рекламации** по проблемному заказу |

---

## Цель работы

Разработать на основе моделирования бизнес-процессов и эмпирической диагностики **комплекс управленческих решений** по совершенствованию цифрового клиентского опыта Dodo Pizza в контуре доставки, направленный на:

- снижение риска недоступности заказа и срыва норматива доставки (отказ при перегрузе ближайшей точки, неясный или нестабильный срок доставки, позднее информирование о задержке);
- повышение эффективности обработки клиентских рекламаций в цифровых каналах.

---

## Задачи

1. Описать организационные и содержательные рамки проекта, цифровой контур Dodo Pizza и предмет исследования.
2. Провести диагностику двух управленческих проблем: недоступность/неуправляемость доставки при перегрузе точки и риске срыва срока, а также неэффективная обработка рекламаций в цифровых каналах.
3. Собрать и проанализировать эмпирические данные: опрос, интервью, вторичный анализ отзывов.
4. Построить иерархию моделей от верхнего уровня к детальному процессу, включая AS IS и TO BE.
5. Обосновать ожидаемый эффект, ограничения и организационные требования к внедрению решений.

---

## Две проблемы и программа решений

### Проблема № 1 — доступность и срок доставки

**Формулировка.** В цифровом канале клиент сталкивается с двумя связанными сбоями контура доставки. Во-первых, при перегрузке ближайшей пиццерии заказ может быть недоступен или отклонён без предложения альтернативной точки обслуживания. Во-вторых, при принятом заказе клиент не всегда получает ясное и стабильное обещание по сроку: чаще видны этапы выполнения, а уведомления о риске задержки работают непредсказуемо. В результате ожидание по времени становится неуправляемым, растут задержки, отмены и негативный опыт.

**Целевые решения (TO BE):**

| Элемент | Содержание | Статус в проекте |
|---------|------------|------------------|
| Ядро | Проверка доступности точки + динамический расчёт/пересчёт срока доставки + дерево решений по уровню риска | ✅ в BPMN TO BE |
| Ветка A | Альтернативная пиццерия при перегрузе ближайшей/обслуживающей точки | ✅ в BPMN TO BE |
| Ветка C | Проактивные push-уведомления при пересчёте задержки после принятия заказа | ✅ в BPMN TO BE |
| Ветка B | Курьер из соседней точки при нехватке локального ресурса | ⏸ решение после этапа 4 (см. ниже) |

**Моделирование:** детальный BPMN AS IS + TO BE в гл. 3 ([[03_РАНЖИРОВАНИЕ]]).

---

### Проблема № 2 — рекламации

**Формулировка.** После проблемного заказа (опоздание, холодная еда, ошибка в составе) клиент обращается с жалобой. Путь обращения непрозрачен, ответы и компенсации неоднородны, чат-бот слабо понимает запрос — обращение быстро передаётся оператору.

**Целевые решения (TO BE):**

- ИИ-помощник: классификация обращения, краткое резюме, черновик ответа;
- матрица правил по типам жалоб (policy matrix);
- human-in-the-loop: компенсации и сложные случаи утверждает человек;
- улучшение понимания естественного языка в чате до эскалации на оператора.

**Моделирование:** упрощённая TO BE-модель в гл. 3 (не равный по объёму второй BPMN).

## Границы проекта

### In scope — входит в проект

| Область | Что включаем |
|---------|--------------|
| Цифровой клиентский путь | Заказ в приложении/на сайте → проверка доступности → срок/статус доставки → доставка → оценка / рекламация |
| Процессы | Проверка доступности, выбор точки обслуживания и сопровождение срока доставки; обработка клиентских рекламаций |
| Модели | Process Landscape, SIPOC, BPMN 2.0 (полный для P3; упрощённый TO BE для P7), глоссарий |
| Данные | Только **открытые источники** + опрос / интервью / отзывы клиентов (без инсайдеров Dodo) |
| Эмпирика | Опрос 40–60, интервью 8–12, кодирование 80–120 отзывов |
| Решения TO BE | Ядро + ветки A, C; ветка B — по решению после диагностики; ИИ-помощник с контролем человека |

### Out of scope — сознательно не включаем

| Область | Почему исключено |
|---------|------------------|
| Детальная кухня (рецептуры, станции приготовления) | Не предмет цифрового клиентского опыта; влияет на срок, но не моделируется |
| Маркетинг, акции, лояльность | Другой контур |
| HR, найм курьеров, закупки | Вне рамки процесса доставки для клиента |
| Финансы, франчайзинг, юридические отношения | Не нужны для поставленной цели |
| Утверждение «так уже работает у Dodo» для TO BE | TO BE = **предложение** команды, не описание факта |
| Два полноценных BPMN AS IS/TO BE одного объёма | По методичке — один детализированный процесс ([[03_РАНЖИРОВАНИЕ]]) |
| Внутренние KPI и регламенты без публичного источника | Заменяем реконструкцией + [[assumptions_register]] |

## Решения команды (этап 1)

| Вопрос | Решение |
|--------|---------|
| Сколько проблем в курсаче? | Две, связанные одним клиентским путём |
| Какой процесс — главный BPMN? | Доступность и сопровождение срока доставки при риске срыва норматива (процесс № 1) |
| Второй процесс | Рекламации — упрощённая TO BE-модель |
| Ранжирование | Заполнено: [[03_РАНЖИРОВАНИЕ]] |
| **Ветка B (курьер соседней точки)** | **Отложено до этапа 4 (эмпирика).** Если опрос/интервью не подтвердят приемлемость для клиента — вынести в заключение как перспективу, не включать в BPMN TO BE. Если подтвердят — добавить в BPMN на этапе 9. |
| Доступ к данным | Только открытые источники + полевая эмпирика; сотрудников Dodo в команде нет |

## Ограничения проекта

- AS IS процессов **реконструируется** по публичным материалам; спорные места — в [[assumptions_register]].
- Цифры и регламенты Dodo **не выдумываются**; нет источника → `[ТРЕБУЕТСЯ ИСТОЧНИК]` или допущение.
- Эмпирическая выборка **нерепрезентативна** для всей базы клиентов Dodo — ограничение описывается в гл. 2.
